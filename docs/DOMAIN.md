# Main Service — Доменная модель

> Документ описывает сущности, их связи и жизненный цикл для модуля квизов.
> Намеренно не привязан к языку или БД — сначала домен, потом реализация.
>
> Создан: 2026-06-17. Охват: квизы, карточки, черновики. Игровой движок — отдельно, позже.

---

## Глоссарий

| Термин | Значение |
|---|---|
| **Квиз** | Авторский набор карточек. Единица контента платформы. |
| **Черновик** | Рабочая, мутабельная версия квиза. Не видна другим пользователям. |
| **Снапшот** | Зафиксированная копия квиза в момент публикации. Неизменна. |
| **Карточка** | Один вопрос внутри квиза: тип + текст + параметры + ответ. |
| **Вариант ответа** | Один из пунктов выбора в карточке типа SINGLE/MULTIPLE. |

---

## 1. Сущности и связи

### 1.1 Quiz (агрегат)

Корень агрегата. Содержит идентичность квиза и владеет своими частями (черновик, снапшот, их карточки).

**Поля:**
- `id` — уникальный идентификатор
- `ownerId` — идентификатор пользователя-создателя (из auth-service, только ID)
- `visibility` — видимость: `PUBLIC` | `PRIVATE` (по умолчанию `PRIVATE`)
- `draft` — текущий черновик (может отсутствовать)
- `published` — опубликованный снапшот (может отсутствовать)

**Инварианты:**
- У квиза есть хотя бы одно из: `draft` или `published` (оба отсутствовать не могут). Поддерживается приложением — на уровне БД не выражается, т.к. части лежат в отдельных таблицах.
- `visibility` имеет смысл только для опубликованной версии (для UNPUBLISHED фактически не работает).

> **Замечание о направлении связей.** В домене агрегат «владеет» частями. В БД (раздел 4) ссылка идёт в обратную сторону — дети (`draft`, `snapshot`, `card`) ссылаются на родителя через FK. Это деталь персистентности: она даёт каскадное удаление и не меняет доменную картину «агрегат владеет составом».

---

### 1.2 QuizDraft (черновик)

Часть агрегата Quiz. Не существует без Quiz.

**Поля:**
- `title` — название квиза
- `description` — описание (необязательно)
- `cards` — карточки черновика, упорядоченные по `position`
- `createdAt` — дата создания черновика
- `modifiedAt` — дата последнего изменения

**Характеристики:**
- Полностью мутабелен: можно менять название, описание, добавлять/удалять/переставлять карточки
- Не виден другим пользователям и недоступен для запуска игр
- При публикации содержимое переносится в снапшот, черновик удаляется

---

### 1.3 QuizSnapshot (опубликованный снапшот)

Часть агрегата Quiz. Не существует без Quiz.

**Поля:**
- `title` — название на момент публикации
- `description` — описание на момент публикации
- `cards` — карточки снапшота, упорядоченные по `position` (заморожены)
- `createdAt` — дата создания снапшота
- `modifiedAt` — дата последнего изменения (последней публикации)

**Характеристики:**
- Неизменен в обычной работе: правки черновика его не затрагивают.
- Является «живой» версией квиза: виден в каталоге (если PUBLIC), доступен для игр.
- **Редактирование опубликованного квиза:** создаётся черновик-копия снапшота → пользователь меняет черновик как хочет → при повторной публикации **тот же снапшот обновляется** на основе черновика (строка снапшота и её `id` сохраняются, меняется содержимое и `modifiedAt`). Стабильность `id` важна: внешние ссылки на снапшот не ломаются.

---

### 1.4 Card (карточка)

Принадлежит ровно одному родителю — QuizDraft **или** QuizSnapshot. Не существует вне квиза.

**Общие поля:**
- `id` — идентификатор
- `position` — порядковый номер внутри своего родителя (определяет последовательность; нумерация с 0)
- `type` — тип (enum): `SINGLE_CHOICE` | `MULTIPLE_CHOICE` | `TEXT_INPUT`
- `questionText` — текст вопроса
- `timerSeconds` — время на ответ в секундах

**По типу карточки:**

| Тип | Дополнительные данные | Инварианты |
|---|---|---|
| `SINGLE_CHOICE` | Список `CardOption` | 2–8 вариантов; ровно 1 помечен как правильный |
| `MULTIPLE_CHOICE` | Список `CardOption` | 2–8 вариантов; хотя бы 1 помечен как правильный |
| `TEXT_INPUT` | Список допустимых строк | Минимум 1 допустимый ответ |

Карточка живёт строго под одним родителем: либо у черновика, либо у снапшота. При переходах (`startEditing`, `publish`) карточки **копируются с новыми id** — общих карточек у черновика и снапшота не бывает, иначе правка черновика затронула бы снапшот.

---

### 1.5 CardOption (вариант ответа)

Вложен в Card типа SINGLE_CHOICE или MULTIPLE_CHOICE. Не существует без Card. Хранится денормализованно внутри карточки.

**Поля:**
- `id` — идентификатор (понадобится в игре, чтобы принимать ответ по id)
- `text` — текст варианта
- `isCorrect` — правильный ли этот вариант

---

### 1.6 AcceptedAnswer (допустимый ответ для TEXT_INPUT)

Вложен в Card типа TEXT_INPUT. Одна строка = один допустимый вариант написания ответа. Хранится денормализованно (просто список строк, без отдельных id).

**Поле:**
- `text` — допустимый ответ

**Примечание:** при проверке ответа игрока сравнение ведётся без учёта регистра и пробелов по краям.

## 2. Диаграмма связей

```
Quiz  (agg root)
├── draft: QuizDraft?
│   └── cards: List<Card>            (упорядочены по position)
│       ├── Card (SINGLE_CHOICE)   → options: List<CardOption>
│       ├── Card (MULTIPLE_CHOICE) → options: List<CardOption>
│       └── Card (TEXT_INPUT)      → acceptedAnswers: List<AcceptedAnswer>
│
└── published: QuizSnapshot?
    └── cards: List<Card>            (заморожены, та же структура)
```

Всё — **композиция**: карточка не существует без своего черновика/снапшота, вариант ответа — без карточки. Удаление квиза удаляет всё вложенное (в БД — каскадом).

## 3. Жизненный цикл квиза

### Состояния (выводятся из наличия draft/published)

| Состояние | draft | published | Смысл |
|---|---|---|---|
| `UNPUBLISHED` | есть | нет | Новый квиз, ни разу не публиковался |
| `PUBLISHED` | нет | есть | Опубликован, нет незакоммиченных правок |
| `PUBLISHED_WITH_DRAFT` | есть | есть | Опубликован + есть редактируемый черновик |

> **API отдаёт только 2 статуса** — `UNPUBLISHED` / `PUBLISHED` (= есть ли снапшот). `PUBLISHED_WITH_DRAFT` здесь используется лишь как **внутреннее** имя комбинации «есть и draft, и snapshot» для описания переходов; наружу оно не выставляется. Бэкенд в логике опирается на наличие `draft`/`snapshot`, а не на enum. Наличие незакоммиченных правок фронт видит по полю `draft` (на `Quiz`) или по `version` (в `QuizSummary`).

### Переходы

```
[Создать квиз]
      │
      ▼
 UNPUBLISHED ──────[Редактировать]──────▶ UNPUBLISHED
      │
      │ [Опубликовать] (требует ≥1 карточки, все инварианты карточек соблюдены)
      ▼
  PUBLISHED ◀────[Отменить правки]────┐
      │                               │
      │ [Начать редактирование]        │
      ▼                               │
PUBLISHED_WITH_DRAFT ──[Редактировать]──▶ PUBLISHED_WITH_DRAFT
      │
      │ [Опубликовать черновик]
      ▼
  PUBLISHED
```

### Описание операций

| Операция | Доступна в | Результат |
|---|---|---|
| `create` | — | Quiz + пустой черновик создаются вместе (одна транзакция). Состояние `UNPUBLISHED` |
| `editDraft` | `UNPUBLISHED`, `PUBLISHED_WITH_DRAFT` | Обновляет поля и карточки черновика, `modifiedAt = now` |
| `publish` | `UNPUBLISHED`, `PUBLISHED_WITH_DRAFT` | Карточки черновика становятся карточками снапшота; черновик удаляется → `PUBLISHED`. Детали ниже |
| `startEditing` | `PUBLISHED` | Создаёт черновик-копию снапшота (карточки копируются с новыми id) → `PUBLISHED_WITH_DRAFT` |
| `discardDraft` | `PUBLISHED_WITH_DRAFT` | Черновик и его карточки удаляются; снапшот не тронут → `PUBLISHED` |
| `changeVisibility` | любое | Меняет PUBLIC ↔ PRIVATE у квиза; контент не тронут |
| `delete` | любое | `DELETE` квиза → каскадом уходят черновик, снапшот и все карточки |

### Детали `publish`

- **Из `UNPUBLISHED`** (снапшота ещё нет): создаётся новая строка снапшота; карточки черновика переносятся под снапшот; черновик удаляется.
- **Из `PUBLISHED_WITH_DRAFT`** (снапшот уже есть): **существующая** строка снапшота обновляется (`title`, `description`, `modifiedAt`), её `id` сохраняется; старые карточки снапшота удаляются; карточки черновика переносятся под снапшот; черновик удаляется.
- «Перенести карточку под снапшот» = переключить родителя (`draft → snapshot`), сохранив `position`. Поскольку старые карточки снапшота уже удалены, конфликта позиций нет.
- Требование к публикации: в черновике ≥1 карточка и все инварианты карточек соблюдены.

## 4. Схема базы данных (SQL)

Направление связей: **дети ссылаются на родителя** (`draft/snapshot → quiz`, `card → draft|snapshot`), всё с `ON DELETE CASCADE`. Удаление квиза автоматически вычищает весь состав.

```sql
-- Enum-типы
CREATE TYPE visibility AS ENUM ('PUBLIC', 'PRIVATE');
CREATE TYPE card_type  AS ENUM ('SINGLE_CHOICE', 'MULTIPLE_CHOICE', 'TEXT_INPUT');

-- Агрегат: идентичность и видимость
CREATE TABLE quizzes (
    id         UUID PRIMARY KEY,
    owner_id   UUID NOT NULL,                       -- пользователь из auth-service; FK намеренно нет (граница сервиса)
    visibility visibility NOT NULL DEFAULT 'PRIVATE'
);

-- Черновик: максимум один на квиз
CREATE TABLE quiz_drafts (
    id          UUID PRIMARY KEY,
    quiz_id     UUID NOT NULL UNIQUE REFERENCES quizzes(id) ON DELETE CASCADE,
    title       VARCHAR(200) NOT NULL,
    description VARCHAR(1000),
    created_at  TIMESTAMPTZ NOT NULL,
    modified_at TIMESTAMPTZ NOT NULL
);

-- Снапшот: максимум один на квиз
CREATE TABLE quiz_snapshots (
    id           UUID PRIMARY KEY,
    quiz_id      UUID NOT NULL UNIQUE REFERENCES quizzes(id) ON DELETE CASCADE,
    title        VARCHAR(200) NOT NULL,
    description  VARCHAR(1000),
    created_at   TIMESTAMPTZ NOT NULL,
    modified_at  TIMESTAMPTZ NOT NULL
);

-- Карточки: ровно один родитель (черновик ИЛИ снапшот); ответы денормализованы
CREATE TABLE cards (
    id             UUID PRIMARY KEY,
    draft_id       UUID REFERENCES quiz_drafts(id)    ON DELETE CASCADE,
    snapshot_id    UUID REFERENCES quiz_snapshots(id) ON DELETE CASCADE,
    position       INT NOT NULL CHECK (position >= 0),
    type           card_type NOT NULL,
    question_text  VARCHAR(500) NOT NULL,
    timer_seconds  INT NOT NULL CHECK (timer_seconds > 0),
    options        JSONB,        -- [{"id":"...","text":"...","isCorrect":true}, ...]  для SINGLE/MULTIPLE
    accepted_texts TEXT[],       -- ["ответ 1","ответ 2"]                              для TEXT_INPUT

    -- ровно один родитель
    CONSTRAINT one_parent CHECK ((draft_id IS NOT NULL) <> (snapshot_id IS NOT NULL)),

    -- данные ответа соответствуют типу карточки
    CONSTRAINT answer_matches_type CHECK (
        (type IN ('SINGLE_CHOICE','MULTIPLE_CHOICE') AND options IS NOT NULL AND accepted_texts IS NULL)
        OR
        (type = 'TEXT_INPUT' AND accepted_texts IS NOT NULL AND options IS NULL)
    ),

    -- размеры массивов ответов (защита от мусорных значений)
    CONSTRAINT options_max  CHECK (options IS NULL OR jsonb_array_length(options) <= 8),
    CONSTRAINT accepted_max CHECK (accepted_texts IS NULL OR cardinality(accepted_texts) <= 50),

    -- позиция уникальна в пределах родителя; NULL-родитель строки не конфликтуют между собой.
    -- DEFERRABLE — чтобы переупорядочивание карточек в одной транзакции не падало на промежуточных значениях
    CONSTRAINT uq_draft_position    UNIQUE (draft_id, position)    DEFERRABLE INITIALLY IMMEDIATE,
    CONSTRAINT uq_snapshot_position UNIQUE (snapshot_id, position) DEFERRABLE INITIALLY IMMEDIATE
);
```

### Примечания к схеме

- **Каскад сверху вниз:** `DELETE` квиза → уходят его черновик/снапшот → уходят их карточки. Ручного порядка удаления не требуется.
- **Один черновик / один снапшот на квиз** гарантирует `UNIQUE (quiz_id)` на `quiz_drafts` и `quiz_snapshots`.
- **Ровно один родитель у карточки** — `CHECK one_parent` (XOR по null). Общих карточек у черновика и снапшота не бывает.
- **Соответствие типа и данных ответа** — `CHECK answer_matches_type`: для SINGLE/MULTIPLE заполнен `options`, для TEXT_INPUT — `accepted_texts`, лишнее = NULL. Тонкие инварианты (ровно 1 correct, ≥2 варианта) БД не проверяет — это приложение.
- **Уникальность позиций — `DEFERRABLE`:** при перестановке карточек позиции меняются пачкой в одной транзакции, проверка уникальности откладывается до коммита (иначе временный конфликт на промежуточном шаге).
- **`owner_id` без FK** — пользователи живут в auth-service, это межсервисная граница.
- **Денормализация ответов:** `options` (JSONB) и `accepted_texts` (TEXT[]) читаются вместе с карточкой, отдельных таблиц нет — нет N+1.
- **Инвариант «есть draft или snapshot»** на уровне БД не выражается (части в разных таблицах). Держится приложением: `create` создаёт квиз и черновик в одной транзакции.

## 5. Ограничения на стороне приложения

Инварианты, которые БД не проверяет (или проверяет лишь грубо) — их держит сервисный слой.

### 5.1 Карточки

| Тип | Ограничение |
|---|---|
| Все | `questionText` непустой после `trim`; `timerSeconds` > 0 (верхнего предела нет) |
| `SINGLE_CHOICE` | 2–8 вариантов; **ровно 1** с `isCorrect = true`; у каждого варианта непустой `text`; `id` вариантов уникальны в пределах карточки |
| `MULTIPLE_CHOICE` | 2–8 вариантов; **≥1** с `isCorrect = true`; непустой `text`; уникальные `id` |
| `TEXT_INPUT` | ≥1 допустимого ответа; каждый непустой после `trim`; массив не `'{}'` |

> **Жёсткая валидация карточек (5.1) проверяется только при `publish`.** Сохранение черновика (`editDraft`) допускает «сырой», частично невалидный контент — единственное жёсткое требование на сохранение — непустой `title`. Это позволяет фронту сохранять недозаполненный квиз. Детали — в `docs/FRONTEND-BRIEF.md` и контракте API.

- **Структура `options` (JSONB)** валидируется приложением: это массив объектов `{id, text, isCorrect}`, не произвольный JSON. БД проверяет лишь `IS NOT NULL` (см. `answer_matches_type`).
- **`position`** — приложение поддерживает непрерывную последовательность `0..n-1` без дыр при add/remove/reorder. БД гарантирует только уникальность позиции в пределах родителя.

### 5.2 Квиз и публикация

- **`create`** создаёт `quiz` и его `draft` в одной транзакции — иначе нарушится инвариант «у квиза есть draft или snapshot».
- **Число карточек в квизе:** от 1 до 60. Максимум контролируется при добавлении карточки в черновик; минимум (≥1) — при `publish`.
- **`publish`** требует: в черновике ≥1 карточка и все карточки проходят инварианты 5.1.
- **`publish` из `PUBLISHED_WITH_DRAFT`** выполняется строго в порядке: `DELETE` старых карточек снапшота → перенос карточек черновика под снапшот. Так промежуточный конфликт `uq_snapshot_position` не возникает (старые позиции уже удалены к моменту переноса).
- **`editDraft`** обновляет `modified_at`; **`publish`** обновляет `modified_at` — БД сама даты не проставляет.
- **`title`** непустой после `trim`; жёсткого лимита длины пока нет.

### 5.3 Доступ (авторизация)

- Редактирование, публикация, смена видимости, удаление — только владелец (`ownerId` сверяется с `X-User-Id` от gateway).
- **Каталог** отдаёт только квизы с `visibility = PUBLIC` **и** существующим снапшотом.
- Приватный квиз и любой черновик доступны только владельцу.

## 6. Принятые решения и отложенное

### Решено

| Вопрос | Решение |
|---|---|
| Отдельная операция `unpublish` | Не нужна. Убрать квиз из каталога = `changeVisibility` → `PRIVATE` |
| Количество карточек в квизе | От 1 до 60 |
| Число вариантов ответа в карточке | 2–8 |
| Диапазон `timerSeconds` | 1–60 |
| `PATCH /quizzes/{id}` | Меняет **только `visibility`** (уровень квиза). `title`/`description` версионируются и правятся исключительно через сохранение черновика (`editDraft`) — снапшот остаётся неизменяемым |
| Длина текстовых полей | `title` ≤ 200, `description` ≤ 1000, `questionText` ≤ 500, текст варианта/ответа ≤ 500. Заданы и в БД (VARCHAR), и на API (`@Size`) |
| `created_at` | Добавлен на уровне `quiz_drafts` / `quiz_snapshots` (per-версия). Квиз-уровневого нет |
| Оптимистическая блокировка черновика | Не нужна на этом этапе |

### Отложено

- Удаление квиза, по которому уже проводились игры — решаем на этапе игрового движка (снапшот может понадобиться истории игр).
